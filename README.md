# PySpark Joins: All-Pairs Matching & Multi-Way Joins

Αυτό το repository περιέχει την αναλυτική σχεδίαση, υλοποίηση και αποτίμηση δύο θεμελιωδών προβλημάτων στον τομέα των Κατανεμημένων Συστημάτων και της Επεξεργασίας Μεγάλων Δεδομένων (Big Data Analytics), με τη χρήση των **PySpark DataFrames**:

1. **Το πρόβλημα του All-Pairs Matching (`A_part.py`):** Υλοποίηση και βελτιστοποίηση της εύρεσης όλων των πιθανών ζευγών σε ένα dataset, συγκρίνοντας τη Naive προσέγγιση, την SQL προσέγγιση και την Group-based (Afrati et al.) αρχιτεκτονική.
2. **Την Τριαδική Σύνδεση $A(x,y) \bowtie B(y,z) \bowtie C(z,w)$ (`B_part.py`):** Σύγκριση τριών διαφορετικών στρατηγικών για multi-way joins: Απευθείας Τριαδική Σύνδεση (Shares Algorithm / Afrati & Ullman), δύο διαδοχικές δυαδικές συνδέσεις (Sequential API) και αυτόνομο ερώτημα SQL μέσω του Catalyst Optimizer.

## Αρχιτεκτονική & Σχεδιαστικές Επιλογές

Σε όλη την έκταση του κώδικα, αποφύγαμε το παραδοσιακό micromanagement των δεδομένων και εστιάσαμε καθαρά στο **Macro-Programming** και τη δηλωτική (declarative) βελτιστοποίηση της μηχανής του Spark. Συγκεκριμένα, εφαρμόσαμε τις εξής advanced τεχνικές:

* **Zero-Serialization Data Generation:** Αντί να δημιουργήσουμε τις λίστες δεδομένων στην Python (κάτι που θα προκαλούσε τεράστιο serialization overhead στον Driver και κίνδυνο Out-Of-Memory σφαλμάτων), **εμπιστευτήκαμε αποκλειστικά το JVM**. Χρησιμοποιήσαμε τη native εντολή `spark.range()`, αναγκάζοντας τους 16 πυρήνες (workers) να γεννήσουν τα δεδομένα απευθείας κατανεμημένα στη RAM τους, αξιοποιώντας την **Tungsten Engine**.
* **CPU-Level Surrogate Keys:** Στο Σκέλος Α, αντικαταστήσαμε τις δαπανηρές συγκρίσεις χαρακτήρων (String Comparisons) με συγκρίσεις ακεραίων. **Δημιουργήσαμε μοναδικά Integer IDs** μέσω της `F.monotonically_increasing_id()`, επιτρέποντας στην ALU του επεξεργαστή να εκτελεί το τελικό φίλτρο σε μόλις 1 κύκλο ρολογιού αντί για byte-by-byte loops κειμένου.
* **Uniform Bucketing & Load Balancing:** Στον αλγόριθμο Group-based, **επιλέξαμε τη μέθοδο modulo (`% g`)** πάνω στο Integer ID αντί για το κλασικό string hashing. Με αυτόν τον τρόπο, πετύχαμε τέλεια ομοιόμορφη κατανομή των δεδομένων στους κάδους (buckets), εξαλείφοντας το φαινόμενο του **Data Skew** και των καθυστερημένων workers (stragglers).
* **1-Step Multi-Way Joins:** Στο Σκέλος Β, για την απευθείας τριαδική σύνδεση, **στήσαμε ένα διδιάστατο πλέγμα (2D Grid)** από reducers βάσει του Shares Algorithm. Αντιγράψαμε ελεγχόμενα τα δεδομένα των πινάκων $A$ και $C$ μέσω `F.explode()` και `F.sequence()`, ώστε να συναντήσουν τον πίνακα $B$ σε ένα και μόνο βήμα μεταφοράς (1-Step MapReduce), παρακάμπτοντας τη δημιουργία ενός τεράστιου ενδιάμεσου πίνακα.

## Οδηγίες Εκτέλεσης

Για τη δική σας διευκόλυνση, παρέχουμε δύο διαφορετικές επιλογές εκτέλεσης, ανάλογα με το αν επιθυμείτε μια γρήγορη αξιολόγηση των αποτελεσμάτων ή αν θέλετε να πειραματιστείτε με τις παραμέτρους του συστήματος (Scale-Up).

### Επιλογή 1: Αυτοματοποιημένη Εκτέλεση μέσω GitHub Actions (Integration Test)
Για την αποφυγή οποιασδήποτε τοπικής εγκατάστασης ή χρήσης πόρων από το σύστημά σας, έχουμε δημιουργήσει ένα αυτοματοποιημένο CI/CD pipeline. 

Με αυτή την επιλογή, ο κώδικας εκτελείται απευθείας στους απομακρυσμένους Linux servers του GitHub (Ubuntu-latest VMs με 2 Cores / 7GB RAM). Για τον λόγο αυτό, οι τιμές του $N$ στον κώδικα του repository ΘΑ ΠΡΕΠΕΙ ΝΑ ΟΡΙΣΤΟΥΝ συντηρητικά ($N=10.000$ στο Part A και $N=100.000$ στο Part B) προκειμένου το workflow να λειτουργήσει καθαρά ως *Integration Test* (απόδειξη ορθότητας) και να αποφευχθούν timeouts ή OOM (Out-of-Memory) errors στους δωρεάν runners.

**Βήματα:**
1. Μεταβείτε στην καρτέλα **Actions** στο πάνω μέρος αυτού του repository.
2. Στο αριστερό μενού, επιλέξτε το workflow με το όνομα **"Execute PySpark Joins"**.
3. Στα δεξιά, πατήστε το κουμπί **"Run workflow"** και επιβεβαιώστε.
4. Ο server θα αναλάβει αυτόματα να στήσει το Docker περιβάλλον, να τρέξει και τα δύο σκέλη (Α και Β), και μπορείτε να δείτε τους τελικούς χρόνους εκτέλεσης ζωντανά στα logs του browser σας.

### Επιλογή 2: Τοπική Εκτέλεση με Docker (Για Πειραματισμό, HTML Reports & Scale-Up)
Αν διαθέτετε εγκατεστημένο το Docker στο σύστημά σας (είτε μέσω Docker Desktop για Windows και macOS, είτε μέσω της εγγενούς Native Engine για Linux) και επιθυμείτε να πειραματιστείτε με τα πραγματικά όρια της μνήμης ή να εξαγάγετε τα αναλυτικά **HTML Reports**, προτείνεται η τοπική εκτέλεση.

**Βήμα 1: Δημιουργία της Εικόνας (Build)**
Ανοίξτε το τερματικό σας στον ριζικό φάκελο του project (εκεί που βρίσκεται το `Dockerfile`) και εκτελέστε την παρακάτω εντολή για να χτιστεί η εικόνα:
```bash
docker build -t pyspark-assignment .


Βήμα 2: Εκτέλεση των Αλγορίθμων (Run with Volume Mount)
Προκειμένου τα αρχεία αναφορών (report_A.html, report_B.html) να αποθηκευτούν τοπικά στο μηχάνημά σας μετά την εκτέλεση του απομονωμένου container, κάνουμε mount τον τρέχοντα φάκελο χρησιμοποιώντας την παράμετρο -v.

Για την εκτέλεση του Σκέλους Α (All-Pairs Matching):
Bash

docker run --rm -v "${PWD}:/out" pyspark-assignment python A_part.py

Για την εκτέλεση του Σκέλους Β (Triadic Join):
Bash

docker run --rm -v "${PWD}:/out" pyspark-assignment python B_part.py

--rm: Η παράμετρος αυτή είναι κρίσιμη, καθώς δίνει εντολή στο Docker να καταστρέφει αυτόματα το container αμέσως μόλις τερματιστεί η εκτέλεση του PySpark κώδικα. Με αυτόν τον τρόπο, διασφαλίζουμε ότι ο σκληρός σας δίσκος παραμένει καθαρός και δεν γεμίζει με ανενεργά containers μετά από πολλαπλές δοκιμές.

Παραμετροποίηση & Config Tweaking (Κατά την Τοπική Εκτέλεση)

Αν ακολουθήσετε την Επιλογή 2 (Τοπική Εκτέλεση), μπορείτε να ανοίξετε τα αρχεία .py και να πειραματιστείτε με τις παρακάτω μεταβλητές για να προσομοιώσετε συνθήκες Big Data Analytics (Stress Testing):

    N (Μέγεθος Δεδομένων): Μπορείτε να αυξήσετε ελεύθερα το μέγεθος των πινάκων (π.χ. στο A_part.py ελέγξαμε τοπικά N=1.000.000, οδηγώντας με επιτυχία στην παραγωγή σχεδόν 500 Δισεκατομμυρίων ζευγών). Λόγω της εγγενούς (native) παραγωγής δεδομένων απευθείας στο JVM μέσω της spark.range(), η Python δεν εξαντλεί τη μνήμη της.

    spark.sql.shuffle.partitions: Από προεπιλογή το ρυθμίσαμε στο CORES * 2 για βέλτιστο Load Balancing. Αν το αλλάξετε στο προεπιλεγμένο 200 του Spark, θα παρατηρήσετε αισθητή καθυστέρηση σε μικρά datasets λόγω του Task Scheduling overhead.

    spark.sql.autoBroadcastJoinThreshold: Αν προσθέσετε την εντολή .config("spark.sql.autoBroadcastJoinThreshold", "-1") στο Setup, απαγορεύετε ρητά στο Spark να εκτελέσει αστραπιαία Broadcast Joins στη μνήμη. Έτσι, το αναγκάζετε να προσομοιώσει βαριές συνθήκες Network Shuffle (SortMergeJoin).
    (Στο Σκέλος Β, η προσέγγιση SQL της Catalyst Optimizer ενδέχεται να είναι εκπληκτικά γρήγορη ακριβώς επειδή εκμεταλλεύεται by default την L3 Cache του επεξεργαστή μέσω Broadcast. Απενεργοποιώντας το, επιτυγχάνουμε δίκαιη σύγκριση).

Στρατηγικές Κλιμάκωσης (Scale-Up) ανά Λειτουργικό Σύστημα

Σε περίπτωση που αυξήσετε το N σε ακραία επίπεδα (>107 εγγραφών), η διαδικασία καθίσταται έντονα Memory και I/O Bound. Παραθέτουμε συνοπτικά τις ρυθμίσεις δέσμευσης πόρων ανά περιβάλλον για την αποφυγή OOM (Out-of-Memory) σφαλμάτων:

    Σε Linux (Native Docker Engine):
    Το container εκτελείται απευθείας στο host kernel χωρίς ενδιάμεσο Virtualization overhead. Δεν απαιτείται καμία ρύθμιση επιπέδου Docker. Η διαχείριση πόρων γίνεται αποκλειστικά στον κώδικα (στα αρχεία .py), αυξάνοντας τη διαθέσιμη μνήμη στο SparkSession (π.χ. .config("spark.driver.memory", "12g")).

    Σε macOS (Docker Desktop via Virtualization):
    Λόγω της αρχιτεκτονικής του Docker στο macOS (το οποίο βασίζεται σε ένα ελαφρύ Linux VM), οι προεπιλεγμένοι πόροι (συνήθως 2GB RAM) ανεπαρκούν. Απαιτείται ρητή αύξηση της μνήμης (συνίσταται 8GB+) και του Swap file από το UI: Docker Desktop -> Settings -> Resources -> Advanced.

    Σε Windows (Docker Desktop via WSL2):
    Αν και το υποσύστημα WSL2 διαχειρίζεται δυναμικά τη μνήμη, σε περιπτώσεις ακραίου φόρτου (extreme stress tests) προτείνεται ο ρητός ορισμός των ανώτατων ορίων μέσω του αρχείου .wslconfig (π.χ. memory=12GB). Επιπλέον, λόγω του εκτεταμένου Spill to Disk κατά το Shuffle Phase, διασφαλίστε ότι υπάρχει επαρκής ελεύθερος χώρος στο host drive.
